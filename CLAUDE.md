## Project Overview

CS479 course assignment: single-category 3D point cloud instance segmentation of a "Nubzuki" figurine. Synthetic training data is generated by inserting Nubzuki objects (from `../assets/sample.glb`) into real MultiScan indoor scenes. The model is a SoftGroup-inspired architecture using sparse convolutions.

**Hard constraints:** ≤50M parameters, ≤24GB VRAM, ≤300s inference time per test set.

## Common Commands

All commands should be run with the `3d-seg` conda environment:

```bash
# Data generation
conda run -n 3d-seg python yoon/generation.py \
    --multiscan-dir ../assets/multiscan \
    --glb-path ../assets/sample.glb \
    --output-dir yoon/generated_data \
    --n-train 5000 --n-val 200 --n-test 200 --seed 42

# Training
conda run -n 3d-seg python yoon/train.py \
    --data-dir yoon/generated_data \
    --ckpt-dir yoon/checkpoints \
    --epochs 30 --lr 1e-3

# Resume training from checkpoint
conda run -n 3d-seg python yoon/train.py \
    --data-dir yoon/generated_data \
    --ckpt-dir yoon/checkpoints \
    --epochs 30 --resume yoon/checkpoints/best_model.pth

# Evaluation (uses provided script — do not modify evaluate.py)
conda run -n 3d-seg python evaluate.py \
    --test-data-dir yoon/generated_data/test \
    --ckpt-path yoon/checkpoints/best_model.pth \
    --output-dir yoon/eval_out
```

## Architecture

The model pipeline (`model.py`) operates as follows:

1. **Input:** `[B, 9, N]` point features — `xyz` (3) | `rgb` (3) | `normal` (3)
2. **Voxelization:** Points mapped to a `150×150×150` sparse voxel grid (`voxel_size=0.02`, `coord_range=[-1.5, 1.5]³`)
3. **Sparse U-Net Backbone:** 4-level encoder/decoder with SubMConv3d blocks; channel progression `32→64→128→256`, skip connections at each level
4. **Per-Point Heads:** Semantic head (fg/bg classification) + offset head (displacement toward instance center)
5. **Proposal Generation:** `sklearn.cluster.DBSCAN` (`eps=0.02`, `min_samples=20`, `algorithm='ball_tree'`) on offset-shifted foreground points with `fg_threshold=0.5`. After a proposal is accepted, its mask is **spatially expanded**: all points with `fg_prob >= 0.2` within the cluster's bounding sphere (centroid ± max-radius in original xyz) are added to the instance. This two-threshold design separates finding WHERE an instance is (strict 0.5, clean clusters) from determining WHICH points belong to it (loose 0.2 + spatial proximity). Higher-confidence proposals claim points first; already-claimed points are not re-assigned.
6. **Proposal Scoring:** Classification, segmentation, and IoU-prediction heads refine and rank proposals

## Data Format

Generated `.npy` files are dictionaries with:
- `xyz`: `(N, 3) float32` — raw world coordinates
- `rgb`: `(N, 3) uint8` — color `[0, 255]`
- `normal`: `(N, 3) float32` — surface normals
- `instance_labels`: `(N,) int32` — `0=background`, `1..K=object instances`

During `dataset.py` loading: xyz is radius-normalized, rgb is divided to `[0, 1]`, normals are unit-length.

## Training Details

- Proposal loss is only applied after epoch 3 (backbone needs to warm up first)
- Batch size is 1 throughout (VRAM constraint)
- Predictions are capped at 100 instances; higher IDs are remapped to background
- Checkpoints saved to `yoon/checkpoints/best_model.pth` based on best F1@0.25
- **Semantic loss** uses per-batch class weighting: `weight = [1.0, bg_count/fg_count]` to counteract the 5:1–30:1 background/foreground imbalance
- **Proposal backward strategy:** backbone loss (semantic + offset) is backward'd first; proposal heads are trained separately on `point_feats.detach()` with `backward()` called every 4 proposals to bound peak GPU memory. The backbone does not receive gradients from proposal losses.
- **Offset error** is logged each epoch as `offset_err` (mean absolute per-point error over foreground points in the validation set), providing a direct signal on whether offset predictions are converging
- Validation prints: `loss | val_f1@25 | val_f1@50 | offset_err`

## Augmentation

**Geometric (per inserted object):** Anisotropic scaling `Uniform(0.5, 1.5)` per axis; rotation ±180° per axis (ZYX Euler); placement only on upward-facing surfaces (`normal_z > 0.7`).

**Color (per inserted object):** HSV-space jitter — hue replaced with `Uniform(0, 1)` (full colour wheel); saturation and value each shifted by `Uniform(−0.2, +0.2)` and clamped to [0, 1]. RGB channel jitter is deprecated. Implemented in `yoon/generation.py:_color_jitter` via `matplotlib.colors.rgb_to_hsv / hsv_to_rgb`.

## Known Failure Modes & Diagnosis

See `yoon/diagnosis.md` for a full root-cause analysis. Key points:

- **Clustering radius (`eps=0.02`) is fine for connectivity.** The BFS/DBSCAN radius only needs to exceed the inter-point gap within a converged cluster (~0.003–0.007 in normalized space for 1000+ point instances), not the object radius. The radius is NOT the primary bottleneck.
- **Primary failure mode:** If the proposal heads are undertrained (e.g. training crashes before epoch 3), their output confidence scores average ~0.25, which falls below the hardcoded 0.3 threshold in `predict()` — silently rejecting every proposal and producing 0 predictions per scene.
- **Class imbalance** (5:1–30:1 bg/fg) causes the semantic head to predict all background without class weighting, starving the clustering of foreground points.
- **Training validation F1 ≠ official F1.** `train.py` validation reports micro-averaged F1 over generated val data; `evaluate.py` uses per-scene F1 over test data. Expect a gap; trust `evaluate.py`.
- **Offset error metric:** A decreasing `offset_err` each epoch confirms the offset head is learning. If it plateaus early, the backbone is likely undertrained or the class imbalance is preventing fg gradient flow.
